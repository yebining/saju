# 사주 허브 리디자인 — 설계 문서

> 2026-06-03 · 기존 "관계 사주" MVP(2026-05-09 spec)를 **다(多)카테고리 사주 허브 + 따뜻한 UI**로 피벗.

## 1. 배경 & 문제

기존 MVP는 ① UI가 딱딱하고 못생김, ② 첫 화면에 "이게 뭐 하는 서비스인지" 설명이 없음, ③ "관계 궁합" 단일 기능이라 진입 장벽이 높음. 사주 계산 로직(Task 1~5)은 이미 동작하지만, 그 위에 올라가는 제품 컨셉과 UI를 새로 잡는다.

## 2. 컨셉: "따뜻한 AI 사주 친구"

점집의 신비주의가 아니라 **다정한 친구가 정리해주는 나에 대한 인사이트**. 크림·테라코타 톤, 둥근 카드, 큰 글씨, 안심 문구. 입문자·MZ 친화적. (브레인스토밍에서 컨셉 A 채택, 결과 화면에 C의 점수·그래프 요소 결합.)

## 3. 목표 & 비목표

**목표**
- 홈에서 서비스가 뭔지 한 줄로 이해되고, 5개 카테고리 중 골라 들어갈 수 있다.
- 카테고리: **종합 사주 · 연애운 · 재물운 · 건강운 · 관계 궁합**.
- 각 결과는 **점수 게이지 + 오행 밸런스 그래프 + 강점/주의점/조언**으로 일관된 포맷.
- 입력은 생일 중심으로 부드럽게. "시간 몰라도 괜찮아요" 안심 흐름.
- 따뜻한 디자인 시스템으로 전면 교체.

**비목표 (YAGNI / v2 이후)**
- 회원가입·로그인·결제, 공유/OG 이미지, 다국어, 알림.
- 십성·대운·신살 등 고급 명리 분석.
- 데일리 운세(매일 갱신), 운세 히스토리.
- 결과 영구 저장 (현행 7일 TTL 유지).

## 4. 카테고리 정의

| 키 | 이름 | 입력 인원 | 점수 의미 | 비고 |
|---|---|---|---|---|
| `general` | 종합 사주 | 1명 | 전반적 기운의 균형도 | 오행 밸런스가 핵심 근거 |
| `love` | 연애운 | 1명 | 지금 시기의 연애 흐름 | |
| `wealth` | 재물운 | 1명 | 재물 기운 | |
| `health` | 건강운 | 1명 | 건강·체력 기운 | |
| `relationship` | 관계 궁합 | 2명 | 두 사람 합/충/생/극 종합 | 기존 관계 사주 흡수 |

`relationship`만 입력 단계에 "상대" 입력이 1스텝 추가된다. 나머지는 본인 1명.

## 5. 정보 구조 / 라우트

```
/                       홈 — 한 줄 소개 + 카테고리 허브 5종
/check/[category]       입력 — 생일(+관계면 상대까지)
/result/[id]            결과 — 점수·그래프·강점/주의점/조언
/not-found              만료/오류
```

- 만세력(4기둥 표)은 **결과 페이지 안의 접이식 섹션 "내 사주 자세히 보기"**로 흡수 (기존 `/manse/[id]` 독립 라우트는 제거).
- 카테고리는 URL 세그먼트로 구분 → 입력/결과/프롬프트가 카테고리를 공유.

## 6. 디자인 시스템 (전면 교체)

기존 다크 네이비+골드 토큰을 따뜻한 톤으로 교체한다. `globals.css`의 `@theme` 블록 갱신 (Tailwind v4 CSS-first 유지).

```
--color-bg:     #FBF3E9   /* 크림 */
--color-bg2:    #F7E7D8   /* 그라데이션 하단 */
--color-fg:     #4A3B30   /* 진한 브라운 텍스트 */
--color-accent: #B5562F   /* 테라코타 (포인트/점수/CTA) */
--color-muted:  #9B8473   /* 보조 텍스트 */
--color-card:   #FFFFFF   /* 카드 */
--color-border: #EBD9C7   /* 부드러운 테두리 */
```

- 오행 색상은 기존 유지: 목 `#7BA77E` · 화 `#C75A4A` · 토 `#B89F6E` · 금 `#A8A8A8` · 수 `#5476A8`.
- 둥근 모서리(14~24px), 부드러운 그림자, 큰 타이포. 폰트는 기존 Noto Serif/Sans 유지하되 본문은 Sans 위주.
- 배경은 크림→피치 세로 그라데이션.

## 7. 화면 명세

### 7.1 홈 (`/`)
- 인사 + 한 줄 소개("생일만 알려주면 AI가 당신의 사주를 풀어드려요").
- 카테고리 카드 그리드(2열, 관계 궁합은 가로 1줄). 각 카드: 이모지 + 이름.
- 카드 클릭 → `/check/[category]`.
- 기존 create-next-app 기본 `page.tsx`를 완전히 교체.

### 7.2 입력 (`/check/[category]`)
- 큰 제목("생일을 알려주세요 🎂") + 양/음력 **탭 토글**.
- 생년월일·태어난 시간 필드(탭하면 피커). "태어난 시간을 몰라요" 체크 → 시주 null 처리(기존 로직 그대로).
- `relationship`이면 1스텝(나) → 2스텝(상대) 진행 후 제출. 그 외 1스텝.
- 제출 → API 호출 → 로딩 → `/result/[id]`.
- 기존 `input-form.tsx`를 이 디자인/플로우로 리워크(3-step 관계 전용 → 카테고리 기반 1~2 step).

### 7.3 로딩
- 제출 직후 "사주를 풀어보는 중…" 따뜻한 로딩 화면(최대 ~30초 안내). 기존 흐름과 동일하게 입력측에서 대기.

### 7.4 결과 (`/result/[id]`)
순서: **카테고리 칩 → 점수 게이지(원형, 0~100 + 라벨) → 한 줄 헤드라인 → 오행 밸런스 막대 그래프 → 강점 / 주의점 / 한 줄 조언 → "내 사주 자세히 보기"(만세력 접이식) → 면책 문구**.
- 관계 궁합이면 오행 그래프를 "나/상대" 2개 노출 + 두 일간 관계 한 줄(기존 `relations.ts` 재사용).
- 시 미상이면 상단에 안내 배지.

## 8. 데이터 & 백엔드

기존 2026-05-09 plan의 Task 6~9 백엔드 골격을 **카테고리 파라미터화**하여 재사용한다.

- **계산**: `lib/saju/{data,calculate,relations}.ts` 그대로 사용. 입력 타입에 `category` 추가.
- **저장 (Supabase)**: `results` 테이블에 `category text` 컬럼 추가. 그 외(7일 TTL, RLS read-only, service_role insert) 동일.
- **AI 풀이 (Claude API, Sonnet 4.6 + tool use + zod)**:
  - 결과 스키마 공통화: `score`(0~100), `score_label`, `headline`, `ohaeng_note`, `strengths`(2~3), `cautions`(1~2), `advice`(한 줄). 관계 궁합은 추가로 `relation_line`.
  - **카테고리별 시스템 프롬프트**: `prompts.ts`에 카테고리 → 프롬프트 맵. 공통 명리 규칙 + 카테고리별 관점(연애/재물/건강/종합/궁합)을 주입.
  - 점수 산정 기준을 카테고리별로 프롬프트에 명시(예: 종합=오행 균형, 연애=일간/지지 관계 등).
- **API route** `POST /api/generate-result`: body에 `{ category, me, them?, context? }`. 계산 → insert(pending) → Claude 호출 → update(ready) → `{ id }` 반환. 기존 라우트 일반화.

## 10. 에러 처리

- AI가 tool 미호출/스키마 불일치 → 1회 재시도 후 실패 시 사용자 친화 에러("잠시 후 다시 시도해주세요").
- 잘못된/만료된 `id` → `/not-found`.
- 입력 검증: 날짜 범위, 음력 윤달 토글은 기존 로직 유지.

## 13. 단계 제안 (구현 플랜에서 구체화)

1. 디자인 시스템 교체(토큰·레이아웃) → 즉시 "예뻐짐" 체감.
2. 홈 허브 + 카테고리 라우팅.
3. 입력 화면 리워크(카테고리 기반).
4. 결과 화면(게이지·오행·접이식 만세력) — AI 없이 더미 데이터로 먼저.
5. Supabase + 카테고리별 Claude API 연결(외부 키 필요).
6. 배포.

1~4는 외부 키 없이 진행 가능(가장 큰 시각적 개선). 5~6에서 키 필요.
